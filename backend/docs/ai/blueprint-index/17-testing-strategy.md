# Blueprint §17 — Testing strategy and quality gates

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §17.1–17.8

- Eleven test layers, each with a required cadence — unit/domain and repository/API integration on every PR, contract on PR/staging, E2E on staging/release, concurrency/property PR-critical+nightly, load/soak at preprod/release cadence, security PR-subset+release-full, migration on PR/release, AI eval/red-team on prompt/model release, DR/chaos on a schedule (§17.1).
- Coverage policy: line coverage alone is never proof of quality; domain calculation/state/policy targets ≥90% branch coverage, overall changed code ≥80% with an exception review process; every production bug needs a regression test where technically feasible (§17.2).
- Test data: factories build tenant A/B plus the full domain graph (roles, customer, catalog, inventory, conversation, order); synthetic data only, no production data; time/UUID/provider clients are injectable for determinism; golden fixtures are versioned/checksummed (§17.3).
- Mandatory critical suites (§17.4): auth — refresh rotation/reuse/family revoke, MFA/recovery-code replay, tenant-switch on inactive membership (§17.4.1); multi-tenant — cross-tenant CRUD/list/search/export, nested IDs/composite FK, cache/search/vector isolation, forged-tenant queue messages (§17.4.2); inventory/order concurrency — 100 parallel reservations for last units, multi-SKU lock order/deadlock, reserve-vs-expire-vs-confirm race, double cancel/confirm/retry (§17.4.3); webhook/message — valid/invalid signature, duplicate/out-of-order events, DLQ/reprocess produces exactly one business effect (§17.4.4); payment — duplicate callback, amount/currency mismatch, retry never double-credits/refunds (§17.4.5); AI — prompt injection, cross-tenant retrieval/tool use, stale-evidence claims, unauthorized discount/order/refund, kill switch mid-flight (§17.4.6).
- Property-based invariants: order total = sum of line allocations + shipping + tax + fee − discount; money is integer minor units with no overflow; inventory `reserved/on_hand/blocked/damaged >= 0`; reserve/release/convert sequences never create/destroy quantity outside movement semantics; the state machine rejects every unspecified transition; an idempotent command applied N times equals applied once (§17.5).
- Load scenarios: baseline mixed traffic at capacity target, webhook burst 500 RPS/5m, 20k SSE connections plus reconnect storm, 100 order-confirms/s on a hot SKU, AI 200 concurrent requests with tenant fairness/cost limits, large import concurrent with normal traffic, 4–8 hour soak for leaks (§17.6).
- Migration tests: fresh-DB apply from zero, upgrade snapshot from a production-compatible schema, old code still works after an expand migration mid-rollout, new code tolerates pre-backfill data, backfill is resumable/idempotent (§17.7).
- Release acceptance evidence bundles commit/image digest/SBOM/signature, migration results, OpenAPI/AsyncAPI diff, all test-tier results, AI eval result if affected, load/smoke result by risk, known issues/risk acceptance, and a rollback plan/owner (§17.8).
