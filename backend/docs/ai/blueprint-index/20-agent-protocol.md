# Blueprint §20 — Backend AI Agent operating protocol

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §20.1–20.5

- Read order before coding (§20.1): ticket/acceptance criteria → relevant blueprint sections → the latest ADR → OpenAPI/AsyncAPI/error/permission catalogs → existing module public API and migrations → existing tests/generated artifacts. Existing code never outranks an invariant/security contract.
- Per-ticket flow (§20.2): Step 1 Preflight (confirm DoR, list affected modules/files/contracts/tables/events, run baseline tests) → Step 2 Contract first (OpenAPI/AsyncAPI/error/permission changes before code, lint + breaking-change check + examples/mock) → Step 3 Test design (happy path, validation/permission/tenant negative, idempotency/retry, state conflict, transaction rollback, concurrency if critical, telemetry/audit assertions) → Step 4 Data change (new migration only, constraint/index/RLS/composite FK, fresh+upgrade test) → Step 5 Domain/application implementation (framework-free domain, external calls outside the DB transaction) → Step 6 Cross-cutting controls (authz/field filtering, idempotency, audit/outbox, logs/traces/metrics, feature flag/quota, error mapping) → Step 7 Verification (tests, generated client compiles, log PII/secret check, rollback/old-code compatibility) → Step 8 PR completion manifest (fixed template: Ticket / Behavior changed / Contracts changed / DB migration / Permissions / Transaction-idempotency / Audit-events / Telemetry / Tests-evidence / Feature flag-rollout / Rollback / Known risks).
- Ambiguity resolution order (§20.3): find the rule in blueprint/ADR/contract/metric dictionary → apply the documented safe default → if non-P0 and still unresolved, ship behind a feature flag and log a proposed-ADR/open question to `docs/collaboration/SIGNOFF_TRACKER.md` → if it touches money, inventory, permission, tenant, privacy, payment, AI high-risk action, or data deletion, stop that mutation at a safe interface/mock until the Human Owner decides via the decision queue. No silently invented business rules and no silent `TODO` on the critical path.
- Forbidden behaviors (§20.4): disabling/skipping tests, RLS, permission checks, audit, or scans to make CI pass; `any`, dynamic SQL/string interpolation, plaintext secrets; adding an endpoint without an OpenAPI/permission/error contract; generic PATCH on critical status; calling an external provider inside a transaction; retrying a non-idempotent mutation; writing directly to another module's table or letting AI write business DB directly; logging request body/token/message/PII by default; editing a migration that already ran; self-activating an AI model/prompt/autopilot; using production data in tests/local.
- Self-review checklist (§20.5), run as a separate pass before marking a ticket done — invariant/state transition at the right layer, tenant-context spoofing via cache/search/event/file, lock/idempotency under concurrency/retry/crash, dual-write via outbox/inbox, permission/field-level/error-existence leak, money/timezone/rounding/snapshot correctness, external timeout/retry/circuit-breaker, audit/telemetry sufficiency without leaking data, migration rolling/lock/rollback safety, contract backward compatibility, and negative/concurrency test coverage. The Human Owner may spot-check this exact checklist on any ticket touching money, tenant isolation, permission, or security.
