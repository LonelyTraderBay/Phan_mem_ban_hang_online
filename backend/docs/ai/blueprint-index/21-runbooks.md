# Blueprint §21 — Operational runbooks (minimum required logic)

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §21.1–21.10

- Real runbooks live in `infra/runbooks/`; this section is the mandatory decision logic Backend AI Agent executes for technical triage. Once real paying-tenant traffic exists, any customer-impacting recovery action (data restore, force-revoke, compensating financial/inventory adjustment) MUST get Human Owner approval before execution — it is never automated.
- API 5xx/SLO burn (§21.1): confirm blast radius (env/tenant/route/version/dependency) → correlate with recent deploy/migration/flag change → rollback to the prior image digest if release-related, or activate circuit breaker/degraded mode if dependency-related → protect data correctness over availability (disable mutation if consistency is uncertain) → verify synthetic critical journeys before declaring recovered.
- PostgreSQL saturation/deadlock/replication (§21.2): check connection/pool wait, long transactions, lock graph, replication lag → throttle import/backfill/report jobs first, never kill a random critical transaction → route report traffic to a replica → raising the connection limit is never the first fix on an already-saturated DB.
- Queue backlog/DLQ (§21.3): identify queue/oldest-age/failure-class/concentration → pause a poison producer/consumer causing a retry storm → reprocess via the ops API in small batches keeping the same event/job ID → verify no duplicate order/payment/message effect before draining fully.
- Webhook/provider incident (§21.4): never bypass signature verification on a spike; honor backoff/circuit-breaker on provider outage/429; mark the channel `action_required` on token expiry instead of blind retry; after recovery, replay raw events idempotently and reconcile message counts.
- Inventory discrepancy/negative invariant (§21.5, P0/P1): disable reservation/confirm for the affected tenant/SKU via a scoped flag → snapshot balances/reservations/movements/orders as evidence → never hand-edit a balance with ad-hoc SQL → run reconciliation to locate the first divergence → any repair/compensating movement needs Human Owner approval before it touches a real customer's balance.
- Payment/order mismatch (§21.6): freeze automatic fulfillment/refund for the affected order/provider → preserve provider event evidence/signature → repair only via a reconciliation/repair command, never a manual status update → any compensating financial adjustment needs Human Owner approval before execution.
- AI unsafe output/excessive agency (§21.7): activate the most scoped sufficient kill switch (prompt/tool/tenant/provider/global) → cancel/recheck pending auto-send jobs → preserve redacted AI logs with prompt/model/tool/policy versions → add the case to the frozen eval/red-team set and prefer a deterministic policy fix over a prompt-only mitigation → re-enable canary only after explicit Human Owner approval.
- Suspected tenant data leak/security incident (§21.8, always P0): stop the affected endpoint/feature/credential → preserve logs/audit/traces without broadening exposure → revoke sessions/tokens/keys → patch RLS/authorization/cache/index and test sibling resources → the Human Owner owns any legal/customer-notification decision, never automated.
- Failed deploy/migration (§21.9): rollback by image digest if schema-compatible; migrations must be idempotent/resumable — never manually mark a migration successful without verification; a destructive/contract-phase failure follows a prewritten reverse/forward-fix plan.
- Disaster restore (§21.10): Backend AI Agent proposes the recovery point and expected data-loss window; Human Owner approves before restore executes; reconcile provider callbacks/outbox since the recovery point to avoid duplicate external sends; document actual RPO/RTO afterward.
