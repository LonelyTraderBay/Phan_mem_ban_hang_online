# Blueprint §14 — Analytics, metric dictionary and reconciliation

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §14.1–14.6

- Analytics is a derived read model that never mutates the transactional source; every metric needs a business definition, source, event time, timezone, metric version, and reconciliation query. Dashboards must show `data_freshness_at`/watermark, and permission applies to both the report API and export (§14.1).
- Time semantics distinguish `occurred_at` (business event), `ingested_at`, `business_date` (tenant timezone), and `processed_at`; late events revise the business-date fact within a 7-day window, then need a backfill/reconciliation job marked "revised" (§14.2).
- Metric dictionary v1 (§14.3): order metrics — `orders_created/confirmed/cancelled/completed`, `average_order_value` (§14.3.1); revenue — `gross_sales_minor`, `net_sales_minor`, `recognized_revenue_minor` (v1 basis = net item sales of `completed` orders), `refund_minor` (§14.3.2); gross profit = recognized net item sales − locked COGS snapshot from `order_items.unit_cost_minor`, gated by `report.profit.read` (§14.3.3); conversation/SLA — `first_response_seconds`, `resolution_seconds`, `sla_breaches`, `conversation_to_order_rate`, `hot_lead_conversion_rate` (§14.3.4); AI quality — acceptance rate, edit rate/distance, critical rule violations, cost per suggestion (§14.3.5). Never use a bare word like "revenue" without stating gross/net/recognized/paid basis.
- Projection architecture: domain event → `analytics.project` consumer → inbox dedupe by event ID → atomic fact/read-model update + watermark, source event link stored, rebuildable from the event log for a given period (§14.4).
- Daily reconciliation after tenant day-close compares confirmed/completed order counts and amounts, payment sums/refunds, locked cost, conversion facts, and AI usage against source; any monetary/order-count mismatch raises an alert before a report can be considered "closed" (§14.5).
- Report query controls: a max date range for sync endpoints (large ranges go through an export job), allowlisted filter dimensions with matching indexes, a separate statement timeout for the read role, and no ad-hoc heavy aggregate on the primary transactional DB during peak (§14.6).
