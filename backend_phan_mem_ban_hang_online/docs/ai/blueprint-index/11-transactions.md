# Blueprint §11 — Transaction, locking and concurrency

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §11.1–11.11

- Default isolation is `READ COMMITTED` plus explicit row locks/constraints; `REPEATABLE READ`/`SERIALIZABLE` only for benchmarked flows with a retry handler. Transactions must be short — never call a provider/LLM inside a DB transaction. Lock rows in deterministic order, e.g. `(warehouse_id, variant_id)` or aggregate ID, to cut deadlocks (§11.1).
- Reservation algorithm (§11.2): claim idempotency key → validate owner/items → sort allocations by `warehouse_id, variant_id` → `SELECT balances FOR UPDATE` in that order → per item check `available = on_hand - reserved - blocked - damaged - safety_stock`, all-or-nothing → insert reservation header+items → update balances (`reserved += qty`) → audit + outbox `inventory.reserved`.
- Confirm order transaction (§11.3): lock order and verify expected version/state → recompute totals from authoritative catalog (the client quote is never authoritative) → lock reservation+items and verify owner/active/not-expired/quantity-match → lock balances deterministically → convert reservation (`reserved -=`, `on_hand -=`, append ledger movement) → snapshot price/cost/tax/discount/address → set confirmed + history → audit + outbox `order.confirmed`. No payment/carrier/AI calls inside the transaction; analytics/warehouse/notification happen post-commit async.
- Cancel order transaction (§11.4): lock order → validate cancellation policy against order/payment/fulfillment state → release an active reservation → if inventory was already converted but not shipped and policy allows, append a compensating `cancel_restore` movement (never edit the old movement) → a paid order creates a refund-required workflow, never an assumed-successful refund → audit+outbox; idempotent replay must not double-release/restock/refund.
- Payment callback/manual record (§11.5): verify signature → dedupe provider event → lock payment/order → validate currency/amount/reference/state → insert an immutable payment transaction → recompute `payment_status` from sums → amount mismatch routes to reconciliation, never auto-marked paid.
- Outbound message send (§11.6): the request transaction only creates `outbound_messages=queued`; the worker leases the row, rechecks channel health/AI takeover mode/content policy/stale suggestion data, applies the provider rate limiter, then calls the provider outside the DB transaction with a provider idempotency key before the completion transaction records sent/retry/failed/blocked.
- Webhook normalize/upsert (§11.7): dedupe the raw event by unique key; identity/message upsert by external key; conversation creation/upsert is locked by channel account + external thread ID; a repeated event must not double-count unread/lead-score/order triggers.
- Import apply (§11.8): small/medium imports parse into staging then merge in one transaction with fatal rollback; large imports keep staging immutable and apply in idempotent batches with a generation/version switch for atomic final visibility — no half-visible import that still affects inventory/order.
- Customer merge (§11.9): lock source and target by ID order → revalidate not already merged and same tenant → resolve identity conflicts per the preview checksum → repoint allowed FKs (immutable historical snapshots are not rewritten) → mark source merged, append merge history/audit/outbox → retrying the same key returns the same target/result.
- Deadlock/serialization retry (§11.10): the transaction runner retries only SQLSTATE `40P01`/`40001`, max 3 attempts with jitter; validation/permission/business conflicts are never retried; logged metrics must not contain sensitive SQL parameters.
- Reconciliation jobs (§11.11): minimum set is `inventory_balance_vs_ledger`, `reservation_sum_vs_balance_reserved`, `order_totals_vs_items`, `order_payment_status_vs_payment_sums`, `order_facts_vs_orders`, `outbox_pending_age`. These never auto-repair critical data — they raise a discrepancy record/alert with evidence for an approved repair command.
