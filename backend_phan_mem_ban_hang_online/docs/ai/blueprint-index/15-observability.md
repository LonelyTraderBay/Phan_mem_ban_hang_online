# Blueprint §15 — Observability, SRE and operational readiness

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §15.1–15.8

- Every deployable instruments traces, metrics, and logs via OpenTelemetry/OTLP, with trace context propagated over HTTP, queue jobs, and internal tool calls. Standard attribute set is in §15.1.1 — never put email/phone/message text/prompt/token/full URL query/object payload into attributes or metric labels (§15.1).
- Structured JSON logs carry `request_id`, `correlation_id`, `trace_id`, `tenant_id`, `operation`, `outcome`, `duration_ms`; `debug` is off in production by default; exception logs scrub request body/headers; sampling must never drop audit/security/critical errors (§15.2).
- Metrics: RED for services (rate/error/duration), USE for resources (DB pool/locks/replication lag, Redis, queue depth/oldest-job-age), plus business/reliability signals — order-confirm success/conflict, inventory-insufficient vs technical failure, webhook signature-invalid/dedupe lag, AI blocked/fallback/cost, RLS denial anomaly, outbox oldest-pending-age, reconciliation mismatch (§15.3).
- Required traced critical journeys: webhook→queue→normalize→conversation/message→outbox→SSE; conversation reply→outbound queue→provider; order confirm→inventory conversion→outbox→analytics; AI suggestion→retrieval→model→tool→QC→approval/send. Span names stay low-cardinality — no resource IDs in the span name, no raw prompt in provider spans (§15.4).
- Health endpoints: `/health/live` (process only, no heavy dependency check), `/health/ready` (DB/required dependency + migration compatibility), `/health/startup`, `/internal/health/dependencies` (protected/redacted). Redis/AI/provider degradation doesn't necessarily flip core API readiness — it's capability-scoped graceful degradation (§15.5).
- Ten mandatory dashboards: executive SLO/error-budget, API latency/error by route, PostgreSQL health, queue/outbox/DLQ, webhook/channel/provider health, conversation/realtime delivery, order/inventory/payment critical path, AI latency/cost/quality/safety, security/auth/rate-limit/support-access, backup/restore and deploy health (§15.6).
- Alert table with trigger baselines and severities (§15.7): SLO fast-burn (14.4x over paired 5m/1h) P1/P2; order-confirm 5xx >2%/5m P1; any inventory invariant/reconciliation mismatch P1; tenant-isolation/security signal P0; DB connection saturation >85%+wait P2; queue oldest job >2x SLO P2; DLQ growth P1/P2; outbox pending age >60s P2; backup/PITR failure P1; cert/secret expiry ladder P3/P2.
- On-call readiness gate before production requires owner/escalation contacts, runbook links in every alert, tested deploy/rollback access, a backup restore drill, and a blameless post-incident process. Once real paying-tenant traffic exists, a reachable Human Owner backstop is mandatory for customer/financial-impacting incidents — the AI agent alone can never be the sole on-call (§15.8).
