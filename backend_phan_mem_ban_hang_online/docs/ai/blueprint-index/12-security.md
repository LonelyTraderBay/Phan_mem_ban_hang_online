# Blueprint §12 — Security, privacy and secure SDLC

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §12.1–12.14

- Baseline: OWASP ASVS 5.0 Level 2 everywhere, Level 3 for authentication, tenant isolation, admin/support access, payment/refund, secrets, and audit; OWASP LLMSVS/AISVS for AI; NIST SSDF for the software supply chain. Controls must land as concrete backlog/test evidence, never "per OWASP" hand-waving (§12.1).
- A threat model is required before implementation freeze for: multi-tenant/RLS, auth/refresh/MFA, provider OAuth+webhook, AI/RAG/tool-calling/prompt-injection, payment/refund, object upload/download, support impersonation/break-glass, and CI/CD supply chain (§12.2).
- Data classification: Public / Internal / Confidential (customer, order, message, cost, analytics — TLS + at-rest encryption + RBAC/RLS) / Restricted (password hash, refresh/OAuth token, recovery code, encryption key — secret manager/HSM/KMS only, never logged) (§12.3).
- Cryptography: TLS 1.2+ (prefer 1.3); Argon2id passwords with versioned/benchmarked cost and rehash-on-login; refresh/reset/invite/recovery tokens stored hashed only; PII field encryption is envelope encryption via KMS with key versioning; searchable PII uses a keyed blind index, never an unsalted hash; JWT signing is asymmetric with `kid` rotation (§12.4).
- Secrets never live in git/image layers/frontend bundle/CI logs/real `.env.example`; production secrets come from a secret manager/workload identity at runtime; scanning runs at pre-commit/PR/main and a real leak must be revoked, not just deleted from history (§12.5).
- Input/output security: allowlist validation at the API boundary and domain invariant, parameterized SQL only, no dynamic code execution or shell-from-input, AI/message HTML/Markdown treated as untrusted, SSRF protection on URL fetch/import (scheme allowlist, block private/link-local/metadata ranges), upload MIME/size/checksum/malware-scan/quarantine, CSV/Excel export escapes formula injection (§12.6).
- Multi-tier rate limiting — edge/IP, user/session, tenant, provider account, resource — with `429` returning `Retry-After`; AI/export/import use a concurrency budget, not just RPS (§12.7).
- Tenant/IDOR controls: authorization is checked server-side on every object and nested relation (never rely on an "unguessable UUID"); the four layers are RLS + composite FK + repository context + automated tests; cross-tenant access returns 404 to a normal actor; signed object URLs are tenant-bound before signing (§12.8).
- Audit integrity: append in the same transaction as the critical action; the runtime DB role has no UPDATE/DELETE on audit rows; a checksum/hash chain or export manifest detects tampering; reading/exporting audit is itself audited (§12.9).
- Privacy lifecycle: minimize collection to what sales/support/order/fulfillment actually needs (§12.10.1); export is an async job — permission/recent-MFA check, resolve identities, generate an encrypted archive with a short-TTL signed URL, auto-delete after download (§12.10.2); delete/anonymize never hard-deletes accounting/order/payment/audit data under a retention obligation — it anonymizes direct identifiers, revokes identities/consents, crypto-shreds keys, and must purge search/vector/cache/read models too (§12.10.3).
- Object storage: private buckets with no public access, random tenant-prefixed keys (never raw filename/PII), pre-signed upload TTL 5–15 minutes with MIME/size/checksum limits, download URLs only issued after auth/RLS/permission check (§12.11).
- Supply chain: lockfile + trusted registry, SCA/secret/SAST/IaC scans, SPDX/CycloneDX SBOM, signed OCI images with provenance/attestation, least-privilege OIDC deploy credentials (no long-lived cloud keys), SLSA-aligned build provenance (§12.12).
- Vulnerability SLA: exploited-critical ≤24h mitigation; critical ≤3 days; high ≤14 days; medium ≤60 days; low ≤180 days. Unresolved critical/high blocks a production release unless the Human Owner documents a risk acceptance with expiry (§12.13).
- Security test gate before commercial launch covers auth bypass/token replay, tenant IDOR/RLS on every resource, injection classes, SSRF/upload/path traversal, webhook signature/replay, rate-limit/budget bypass, support-access abuse, AI prompt-injection/tool-authorization/exfiltration, dependency/container/IaC findings, plus DAST and a manual pentest (§12.14).
