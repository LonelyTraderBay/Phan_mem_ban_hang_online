# Blueprint §10 — State machines

**Source:** `backend_doc/01_BACKEND_ENTERPRISE_IMPLEMENTATION_BLUEPRINT_v2.0.md` §10.1–10.12

- Every transition is `current state + command + permission + preconditions => next state + domain changes + audit + outbox events`. Never mutate a status column from a generic repository/controller — transitions live in the aggregate/application handler and need full transition-matrix tests (§10.1).
- Tenant lifecycle: `provisioning → active ⇄ suspended → closed` (§10.2). Membership: `invited → active ⇄ suspended → revoked` (revoked is terminal for that record) (§10.3).
- Import job: `uploaded → analyzing → preview_ready → confirmed → applying → completed|failed`; cancel is only allowed before `applying` starts (§10.4).
- Knowledge: `draft → in_review ⇄ draft → approved → published → archived`; v1 requires ingestion to succeed before retrieval goes active (§10.5).
- Channel account health (`healthy|degraded|action_required|disconnected`) is computed from token/scope/rate-limit/webhook-lag/send-failure signals, never set manually; `degraded` does not erase the credential (§10.6).
- Outbound message: `queued → sending → sent | queued(retry) | blocked | failed`; cancel only allowed before send starts (§10.7).
- Conversation is multi-dimensional, never one enum (§10.8): lifecycle `new → open ⇄ resolved → archived` (§10.8.1); waiting-on `none → customer|staff` driving SLA (§10.8.2); sales stage `none → qualified → order_draft → order_confirmed`, does not auto-resolve the conversation (§10.8.3); AI mode `off → copilot → semi_auto → autopilot`, any active mode → `human_takeover` → release, plus an emergency `off` kill switch from any state (§10.8.4); escalation `normal ⇄ escalated` gated by permission/rule engine (§10.8.5).
- Inventory reservation: `active → released | expired | converted`; convert decrements `reserved` and `on_hand` and appends a ledger movement. Replaying a terminal-state command is idempotent (no double effect); cross-owner or expired conversion is forbidden, v1 conversion is all-or-nothing (§10.9).
- Order has independent dimensions, not one status: `order_status` `draft → pending_customer_confirmation → confirmed → completed`, plus `expired`/`cancelled` branches (§10.10.1); `payment_status` `unpaid → pending → partially_paid → paid → partially_refunded → refunded`, derived from immutable payment/refund sums, not a free status edit (§10.10.2); `fulfillment_status` `unfulfilled → allocated → picking → packed → shipped → delivered` (§10.10.3); `return_status` `requested → approved|rejected → in_transit → received → inspected → completed` (§10.10.4).
- Prompt/version lifecycle: `draft → evaluating → approved → active → retired`; approver must differ from creator when required; emergency rollback keeps audit/incident reference (§10.11).
- AI suggestion: `queued → generating → qc_pending → ready | approval_required | blocked | failed → sent | expired`. `approval_required` never auto-sends; `sent` must link the outbound message; suggestion expires when source/tool data or conversation version goes stale (§10.12).
